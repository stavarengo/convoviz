# Convert js/script.js to TypeScript App

## Context

`js/script.js` is a 1681-line monolithic bookmarklet (`javascript:` IIFE) that exports ChatGPT conversations directly from the browser. It's entirely self-contained: custom ZIP builder, IndexedDB persistence, network layer with auth/retry/backoff, DOM-injected floating panel UI, conversation/project scanner, batch export pipeline, and task list tracker.

The goal is to convert this into a proper TypeScript project in `./js-exporter/` while preserving identical runtime behavior. The final output remains a single `javascript:` IIFE bookmarklet.

## Decisions

| Decision | Choice |
|---|---|
| Deployment target | Bookmarklet (TS compiled + bundled → single IIFE) |
| Module decomposition | Deep nesting with directories per concern |
| Bundler | esbuild (`--bundle --format=iife --minify`) |
| Testing | Vitest only (jsdom + fake-indexeddb for DOM/storage) |
| Package manager | npm |
| Old `js/` directory | Delete only after manual user approval of TS version |
| Migration strategy | Strict 1:1 port — no behavior changes, no refactoring of logic |

## Module Boundaries

### `types.ts`
All shared interfaces and type aliases extracted from the monolith's implicit shapes:
- `ExportState` (the top-level `S` object shape)
- `Settings`, `Progress`, `ScanState`, `RunState`, `Stats`, `Changes`
- `PendingItem`, `DeadItem`, `KfPendingItem`, `KfDeadItem`
- `ProjectInfo`, `FileRef`, `Task`, `TaskStatus`

### `utils/`
Pure functions with zero dependencies on other modules. Directly extracted from the top of `script.js` (lines 5-42).

### `zip/`
Self-contained ZIP building. `crc32.ts` has the table + hash function. `zip-lite.ts` has the ZipLite class (including `dosTimeDate`) and imports `u16`, `u32`, `enc` from `utils/binary.ts`.

### `state/`
State management. `defaults.ts` owns the state shape (defaultState, mergeState). `store.ts` owns IndexedDB/localStorage persistence. `debounce.ts` owns the save-debounce timer.

These modules need access to a shared mutable state reference (`S`) and the store. The entry point (`main.ts`) creates these and passes them through.

### `net/`
Network layer. `net.ts` encapsulates auth token management, fetch with retry/backoff/429 handling, and the download trigger. `sleep.ts` is a standalone utility.

`Net` depends on `state` (reads/writes `S.run.backoffUntil`, `S.run.backoffCount`) and `ui` (calls `UI.setStatus`). These are injected or accessed through a shared context.

### `ui/`
DOM layer. `panel.ts` handles injecting the floating panel, rendering stats, binding event handlers. `task-list.ts` is the in-memory task tracker with its own render method.

### `scan/`
API scanning logic. `conversations.ts`, `projects.ts` handle paginated API calls. `changes.ts` computes diff between scan snapshots. `file-refs.ts` extracts attachment/image references from conversation JSON.

### `export/`
The orchestrator. `exporter.ts` contains `rescan`, `start`, `stop`, `exportOneBatch`, `exportKnowledgeBatch` — the core export pipeline logic.

## Dependency Flow

```
main.ts
  ├── types.ts
  ├── utils/*          (pure, no deps)
  ├── zip/*            (depends on utils/binary)
  ├── state/*          (depends on types, utils)
  ├── net/*            (depends on state, ui for status updates)
  ├── ui/*             (depends on state, types)
  ├── scan/*           (depends on net, state, ui)
  └── export/*         (depends on everything above)
```

The entry point (`main.ts`) is responsible for:
1. Calling `assertOnChatGPT()`
2. Initializing IndexedDB (`initIdb`)
3. Loading state (`Store.load`)
4. Creating the save debouncer
5. Injecting UI
6. Wiring event handlers
7. Exposing `window.__cvz_*` globals
8. Handling the interrupted-run recovery

## Cross-Cutting: `addLog`

`addLog(msg)` is the most-used function in the monolith — called from Net (rate limiting), Scan (progress), Exporter (batch status), and UI (user actions). It writes to `S.logs`, calls `saveDebounce(false)`, and calls `UI.renderLogs()`.

In the TS version, `addLog` is a factory function created in `main.ts` that closes over `S`, `saveDebounce`, and `UI`. It's passed to modules that need it via their constructor or context parameter.

## Shared Mutable State Strategy

The monolith uses a single mutable `S` variable and a single `UI` reference, accessed by closure. In the TS version:

- `main.ts` creates `S` and `UI` instances
- Modules that need them receive them via function parameters or a lightweight context object (including `addLog`)
- No global singletons — the IIFE entry point is the composition root
- The `window.__cvz_*` exports happen only in `main.ts`

## Build Pipeline

### esbuild commands

Production (minified):
```
esbuild src/main.ts --bundle --format=iife --target=es2020 --minify --outfile=dist/script.min.js
```

Development (readable):
```
esbuild src/main.ts --bundle --format=iife --target=es2020 --outfile=dist/script.js
```

Post-build step: prepend `javascript:` to produce the bookmarklet version (`dist/bookmarklet.js`).

### npm scripts
```json
{
  "scripts": {
    "build": "esbuild src/main.ts --bundle --format=iife --target=es2020 --minify --outfile=dist/script.min.js && node -e \"const fs=require('fs');fs.writeFileSync('dist/bookmarklet.js','javascript:'+fs.readFileSync('dist/script.min.js','utf-8'))\"",
    "build:dev": "esbuild src/main.ts --bundle --format=iife --target=es2020 --outfile=dist/script.js",
    "test": "vitest run",
    "test:watch": "vitest",
    "typecheck": "tsc --noEmit"
  }
}
```

## Testing Strategy

### Vitest with jsdom
- `vitest.config.ts` sets `environment: 'jsdom'` for UI tests
- Pure logic tests (utils, zip, state/defaults, scan/changes, scan/file-refs) run in default Node environment
- `fake-indexeddb` package provides IndexedDB polyfill for store tests

### Test philosophy
Since this is a 1:1 port, tests verify identical behavior to the current bookmarklet:
- Same state shape after `defaultState()` and `mergeState()`
- Same CRC-32 output for known inputs
- ZipLite produces valid ZIP with correct entries and CRC-32 checksums
- Same file ref extraction from conversation JSON
- Same change computation logic
- Same task list windowing behavior
- UI panel injects expected DOM structure with correct IDs and attributes

### Dependencies (devDependencies)
- `typescript`
- `esbuild`
- `vitest`
- `jsdom` (via vitest config)
- `fake-indexeddb`

## Migration Order (User Stories)

1. **Project scaffolding** — `package.json`, `tsconfig.json`, `vitest.config.ts`, directory structure
2. **Types** — `types.ts` with all shared interfaces extracted from the monolith's implicit shapes
3. **Utils** — `utils/format.ts`, `utils/sanitize.ts`, `utils/binary.ts` + tests
4. **ZIP** — `zip/crc32.ts`, `zip-lite.ts` + tests
5. **State** — `state/defaults.ts`, `state/store.ts`, `state/debounce.ts` + tests
6. **Net** — `net/sleep.ts`, `net/net.ts` + tests
7. **Scan** — `scan/file-refs.ts`, `scan/changes.ts`, `scan/conversations.ts`, `scan/projects.ts` + tests
8. **TaskList** — `ui/task-list.ts` + tests
9. **UI Panel** — `ui/panel.ts` + tests
10. **Exporter** — `export/exporter.ts` + tests
11. **Entry point** — `main.ts` wiring everything together
12. **Build verification** — confirm bundled IIFE output matches current `script.js` behavior
13. **Cleanup** — delete `js/` directory (gated on user approval)

Each story is independently testable and builds on the previous ones.

## Out of Scope

- No behavior changes (same state shape, same API calls, same UI)
- No refactoring of logic patterns
- No browser extension build
- No new features
- The Python side of convoviz is untouched
