# IDB-Based Export Accumulation & Single Final Zip

## Problem

The current bookmarklet downloads a separate zip file per batch (50 conversations each). A large export (e.g., 500+ conversations) produces 10+ zip files that the user must manually merge before importing into convoviz. This is error-prone and creates a poor user experience.

## Goal

Replace per-batch zip downloads with IndexedDB-based accumulation. Individual conversations and their asset files are stored in IDB as they're exported. Once the export is complete (or on user demand), a single zip file is generated and saved — compatible with convoviz's loader without any post-processing.

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Final output | One big zip | Users get a single file ready for convoviz import |
| Download trigger | Auto on completion + manual "Download" button | Convenience: auto-saves when done; button available mid-export |
| IDB lifetime | Keep until explicit reset | Survives page reloads, allows resume across sessions |
| Final zip method | Streaming via File System Access API (FSAA) with in-memory fallback | Avoids memory pressure for large exports; Firefox gets blob fallback |
| Conversation split | `conversations-NNN.json` (100 per file) | Matches convoviz's `_SPLIT_FILE_RE` pattern for split-file loading |
| Known-file (KF) placement | `kf/<projectName>/` subfolder inside the same zip | Single zip, project assets namespaced by folder |

## Architecture

### New IDB Database: `cvz-export-blobs`

A **separate** IndexedDB database from the existing state database. Two object stores:

#### `conv` store
- **Key**: conversation ID (string)
- **Value**: JSON string of the conversation object (the raw API response)
- One entry per exported conversation

#### `files` store
- **Key**: zip-relative path (string), e.g., `file-abc123.png` or `kf/MyProject/doc.pdf`
- **Value**: `Blob` (binary data of the asset file)
- Asset files (images, attachments) fetched during export

### Why a Separate Database

The existing state IDB (`cvz-state`) stores small JSON-serializable state. Export blobs can be hundreds of megabytes. Keeping them separate means:
- State operations stay fast (no large blob transactions slowing reads)
- `reset()` can clear export data independently from state
- Browser storage eviction policies apply per-database

### IDB Helper Module

```
IDB = {
  _open(name, version, stores) → Promise<IDBDatabase>
  putConv(id, jsonString) → Promise<void>
  putFile(path, blob) → Promise<void>
  getAllConvKeys() → Promise<string[]>
  iterateConvs(callback) → Promise<void>   // cursor-based, memory-safe
  iterateFiles(callback) → Promise<void>   // cursor-based, memory-safe
  totalSize() → Promise<number>            // sum of blob sizes for UI
  clear() → Promise<void>                  // delete both stores
}
```

All methods use short-lived transactions. `iterate*` methods use IDB cursors to avoid loading everything into memory at once.

## Modified Export Flow

### Current Flow (per-batch zip)
```
exportOneBatch() → fetch 50 convos → build ZipLite in memory → Net.download(blob)
```

### New Flow (IDB accumulation)
```
exportOneBatch() → fetch batch convos → for each:
  IDB.putConv(id, json)
  for each asset: fetch blob → IDB.putFile(path, blob)
  mark as exported in S.progress.exported
=> update UI with accumulated size
=> on completion (or button click): generateFinalZip()
```

Key changes to `exportOneBatch()`:
1. Remove `ZipLite` construction and `Net.download()` call
2. After fetching each conversation's JSON, write it to `IDB.putConv(id, jsonString)`
3. After fetching each asset blob, write it to `IDB.putFile(zipPath, blob)`
4. After the batch completes, update `S.progress.exported` as before
5. Call `IDB.totalSize()` to update the "Download (X MB)" button

### Asset Path Strategy

Asset files are stored with their zip-relative path as the IDB key:
- Regular assets: `file-<hash>.<ext>` (flat at root, matching current behavior)
- KF project files: `kf/<projectName>/<filename>`

This means the IDB key IS the final zip entry path — no mapping needed at zip generation time.

## Final Zip Generation

### Primary: File System Access API (Chromium)

```js
async function generateFinalZip() {
  const handle = await showSaveFilePicker({
    suggestedName: 'chatgpt-export.zip',
    types: [{ description: 'ZIP', accept: { 'application/zip': ['.zip'] } }]
  });
  const writable = await handle.createWritable();

  // Stream zip entries directly to disk
  // 1. Build conversations-NNN.json files from conv store
  // 2. Write file store entries as-is
  // 3. Write zip central directory
  await writable.close();
}
```

The streaming zip writer writes local file headers + data directly to the `FileSystemWritableStream`. It accumulates only the central directory entries in memory (small: ~100 bytes per entry). This means a 2GB export uses only a few MB of RAM during zip generation.

### Fallback: In-Memory Blob (Firefox / denied picker)

If `showSaveFilePicker` is unavailable or the user cancels:
1. Build the zip in memory using the existing `ZipLite` approach
2. Trigger download via `URL.createObjectURL()` + click

For very large exports on Firefox, this may hit memory limits. The UI should warn if accumulated size exceeds ~500MB and FSAA is unavailable.

### Zip Internal Structure

```
chatgpt-export.zip
├── conversations-001.json    (conversations 1-100)
├── conversations-002.json    (conversations 101-200)
├── ...
├── file-abc123.png           (asset files, flat)
├── file-def456.jpg
└── kf/
    └── MyProject/
        └── document.pdf
```

This structure is directly compatible with convoviz's `loaders.py`:
- `_find_conversation_files()` matches `conversations-NNN.json` via `_SPLIT_FILE_RE`
- Asset files are siblings at the root level for `assets.py` resolution
- The `kf/` subfolder is ignored by convoviz (no harm)

### Split File Generation

During `generateFinalZip()`:
1. Open a cursor on the `conv` store
2. Accumulate conversation objects into batches of 100
3. For each batch, serialize to JSON and write as `conversations-NNN.json` (1-indexed, zero-padded to 3 digits)

## UI Changes

### "Download" Button

A new button in the UI panel, visible when `IDB.totalSize() > 0`:

```
[Download (12.3 MB)]
```

- Shows accumulated export size
- Clicking triggers `generateFinalZip()`
- Disabled during active export batch (to avoid concurrent IDB reads)
- Auto-triggered when the export completes (all pending items exported)

### Status Updates

The existing status line gains an "Accumulated" counter:
```
Exported: 150 | Accumulated: 42.7 MB | Pending: 350
```

### Reset Behavior

The existing "Reset" action now also calls `IDB.clear()` on the export blob database. The confirmation dialog should mention that accumulated export data will be lost.

## Streaming Zip Writer

A minimal streaming zip writer that writes to a `WritableStream` (or accumulates to a buffer for the fallback path):

```
StreamingZip = {
  constructor(writable)           // WritableStream or BufferSink
  addEntry(path, data) → Promise  // data: Uint8Array or Blob
  finalize() → Promise            // writes central directory + EOCD
}
```

Implementation notes:
- Uses STORE method (no compression), same as existing `ZipLite`
- JSON is already text (low compression ratio); images are already compressed
- STORE keeps the implementation simple and avoids CPU overhead
- Must track each entry's offset, CRC-32, and size for the central directory
- For Blob data (from `files` store), reads in chunks via `blob.stream().getReader()` to avoid materializing the entire blob

## Error Handling

| Scenario | Behavior |
|----------|----------|
| IDB write fails (quota) | Surface error in status, pause export. User can download partial zip and reset. |
| FSAA picker cancelled | Fall back to in-memory blob download |
| Browser lacks IDB | Fatal error at startup — IDB is universal in modern browsers, this is unlikely |
| Partial export (user stops) | Data persists in IDB. User can download what's accumulated or resume later. |
| Page reload mid-export | IDB data survives. On next load, UI shows "Download (X MB)" for accumulated data. State IDB tracks which conversations are exported vs pending. |

## Migration

### From Current Version

On first load of the new version:
- `S.progress.exported` (the `{id: update_time}` map) continues to work as-is
- No IDB export data exists yet, so the "Download" button is hidden
- First export will start accumulating in IDB instead of downloading zips
- No migration of old state needed — the IDB export database simply starts empty

### Version Bump

The bookmarklet version should bump to `cvz-bookmarklet-5.0` to reflect this architectural change.

## Scope & Non-Goals

**In scope:**
- IDB storage of conversations and asset blobs
- Streaming zip generation via FSAA with in-memory fallback
- Split `conversations-NNN.json` format
- KF files in `kf/<projectName>/` subfolder
- "Download" button with size display
- Auto-download on export completion
- Reset clears export IDB

**Not in scope:**
- Compression (DEFLATE) — STORE is sufficient
- Background service worker for offline export — bookmarklet runs in page context
- Cloud upload — local file only
- Incremental zip updates (appending to existing zip) — always regenerate from IDB
