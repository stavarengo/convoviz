# Bookmarklet: Task-Based UI Overhaul

## Context

The Convoviz bookmarklet (`js/script.js`, v3.0, ~1400 lines) now exports regular conversations, custom GPT conversations, project conversations, and project knowledge files. With all these concurrent operations, the current UI — a single status text line and two progress bars — can't adequately convey what's happening.

### Current UI Layout (top to bottom)

1. Header: "Convoviz Direct Export" + close button
2. Version info + storage indicator
3. Stats panel: Exported/Pending/Total/Dead, Projects/KF count, Avg/chat, Avg/batch, ETA, Last stop, Last error, Delta since last scan, Delta pending
4. Batch size input + Rescan button
5. Start/Stop buttons
6. Status text (single line)
7. Main progress bar (green, conversations) + secondary progress bar (purple, KF)
8. Log textarea (160px, scrollable)
9. Reset/Export state buttons

### Problem

- The single status line can only show one thing at a time, but concurrent workers (3 by default) are doing different things simultaneously
- The detailed stats block (Avg/chat, ETA, Delta) takes significant space but is rarely consulted during normal use
- Users can't see at a glance which conversations have been processed, which are in progress, and which failed
- The log textarea serves as both a task record and an error log — these are different concerns mixed together

### Design Source

Based on a whiteboard sketch showing:
- A scrollable **task list** where each operation is a visible line item with its own status
- A compact **header** area with overall progress
- A smaller **log area** at the bottom for detailed messages

### What Changes vs Current UI

| Current Element | New | Notes |
|---|---|---|
| Stats panel (12 lines of Avg/ETA/Delta/etc) | Compact 2-row stats | Only numbers that matter: Exported, Pending, Dead, Projects, KF. Overall progress bar inlined. |
| Single status text line | **Removed** | Replaced by task list — each task IS a status line. |
| Main progress bar (green) | Inlined in stats row | Small bar or percentage next to "Pending". |
| KF progress bar (purple) | Inlined in KF stat | "KF: 5/12" with optional small bar. |

| Batch input + Rescan (own row) | Merged with Start/Stop into one controls row | More compact. |
| — | **NEW: Task list** (~180px, scrollable) | The core addition. |

### Task List Data Model

Each task is an in-memory object (NOT persisted to IndexedDB — tasks are transient UI state for the current session):

```js
{
  id: "conv-{uuid}" | "kf-{fileId}",   // unique key
  type: "conversation" | "knowledge",    // for icon/color
  label: "Chat Title",                   // display name
  projectName: "App Checker" | null,     // project context
  status: "queued" | "active" | "done" | "failed",
  detail: "downloading 2/5 files" | null, // sub-status text
  error: "HTTP 429" | null,              // error message on failure
  startedAt: timestamp | 0,
  completedAt: timestamp | 0
}
```

### Task List Behavior

**Population:** Tasks are NOT created for every pending conversation upfront (that would be 200+ items immediately). Instead:
- **Active tasks** appear when a worker starts processing an item
- **Queued tasks** appear for the next few items in the queue (e.g., next 5-10 after the active ones) to give a sense of what's coming
- **Completed tasks** stay visible for context (capped at last ~30)
- **Failed tasks** always stay visible (until reset)

This "sliding window" approach keeps the list manageable regardless of total count.

**Auto-scroll:** The list auto-scrolls to show the most recent activity, but stops auto-scrolling if the user manually scrolls up (to inspect earlier items). Resumes auto-scroll when user scrolls back to bottom.

**Task states and visual indicators:**
- `queued`: `·` dimmed gray text
- `active`: `⟳` green text, possibly with animated spinner via CSS
- `done`: `✓` muted text (fades slightly so active items stand out)
- `failed`: `✗` red text, always visible

**Sub-status:** Active tasks can show a detail line below them (e.g., "downloading 2/5 files", "fetching conversation detail"). This is a single indented line, not a nested task.

**Project context:** Tasks for project conversations show `[Project Name]` prefix in brackets before the chat title.

### Task List Integration with Existing Code

The task list is a **UI-only layer** on top of the existing export pipeline. No changes to `Exporter`, `exportOneBatch()`, or `exportKnowledgeBatch()` logic. The integration points:

1. **`exportOneBatch` worker loop** (currently sets `UI.setStatus("Fetching: ...")` at each step):
   - Instead of `UI.setStatus(msg)`, call `UI.taskUpdate(id, {status, detail})`
   - When worker starts an item: `UI.taskAdd({id, type, label, projectName, status: "active"})`
   - When file download progresses: `UI.taskUpdate(id, {detail: "downloading 2/5 files"})`
   - When done: `UI.taskUpdate(id, {status: "done"})`
   - When failed: `UI.taskUpdate(id, {status: "failed", error: msg})`

2. **`exportKnowledgeBatch` worker loop**: Same pattern but with `type: "knowledge"` tasks.

3. **Scan phase**: A single "Scanning..." task appears during scan. Not per-conversation.

4. **`setStatus(msg)`**: Kept as a fallback for non-task operations (scan progress, "Building ZIP...", etc.). Displayed as a small text line above or below the task list when there's no active task to show it on.

### Compact Stats Row

Replace the current 12-line stats panel with 2 compact rows:

```
Row 1: Exported: 45 · Pending: 203 · Dead: 2 · [▓▓▓▓▓▓▓░░░]
Row 2: Projects: 21 · KF: 5/12
```

The inline progress bar is a thin visual indicator — not a precise tool. The task list provides the detail.

**What's removed:**
- Avg/chat, Avg/batch — developer metrics, not user-facing value
- ETA — unreliable and misleading (rate limits make it unpredictable)
- Last stop timestamp — visible in log if needed
- Last error — visible in task list (failed items show their error)
- Delta since last scan / Delta pending — useful during development, not for users

**What's kept in the log only:**
- Batch summaries ("Batch done: 50 chats in 2m 15s")
- Detailed error messages
- Scan results ("Rescan done. Total 248, pending 203")

### Controls Row

Merge batch input + rescan + start/stop into one compact row:

```
Batch [50] [Rescan] [Start] [Stop]
```

All on one line. Saves vertical space compared to current two-row layout.

### Log Area

Shrinks from 160px to ~80px. Most operational feedback moves to the task list. The log keeps:
- Batch completion summaries
- Error details
- Scan results
- System messages (rate limit warnings, backoff notifications)

The log remains a `<textarea readonly>` for easy copy-paste of error messages.

### CSS Animations (minimal)

One animation for active tasks — a subtle spinner or pulsing dot. Implemented via CSS `@keyframes` injected as a `<style>` element (same technique bookmarklets use for injected styles):

```css
@keyframes cvz-spin { to { transform: rotate(360deg); } }
.cvz-task-active::before {
  content: "⟳";
  display: inline-block;
  animation: cvz-spin 1s linear infinite;
}
```

This is ~3 lines of CSS. No heavy animations.

### Panel Size

Current: 380px wide, height determined by content (~450px estimated).
New: 380px wide, height similar (~450-480px). The task list (180px) + smaller log (80px) roughly equals the old stats block + log (160px + stats ~100px).

The panel should NOT grow significantly. If the task list has many items, it scrolls internally.

### Version Bump

`cvz-bookmarklet-4.0` — this is a significant UI change.

## What This Design Does NOT Include

- **Drag-and-drop task reordering**: YAGNI. The queue order is determined by the export pipeline.
- **Task filtering/search**: YAGNI. The list is a sliding window, not a full history.
- **Persistent task history**: Tasks are transient UI state. The log and IndexedDB state provide the persistent record.
- **Collapsible project groups**: Nice but adds complexity. The `[Project Name]` prefix is sufficient.
- **Multiple simultaneous panels**: One panel, one export session.
- **Dark/light theme toggle**: YAGNI. The dark theme matches ChatGPT's UI.

## Suggested User Stories

**US-001: Task list data model and rendering**
- In-memory task array with add/update/remove operations
- Sliding window: active + recent completed (~30) + failed (all) + next queued (~10)
- HTML rendering with status icons (·, ⟳, ✓, ✗) and project context
- Auto-scroll behavior (follows new items, stops when user scrolls up)

**US-002: Integrate task list with export pipeline**
- `exportOneBatch` worker calls `UI.taskAdd/taskUpdate` instead of `UI.setStatus`
- `exportKnowledgeBatch` worker does the same for knowledge file tasks
- Scan phase shows a single "Scanning..." task
- `setStatus` kept as fallback for non-task status messages

**US-003: Compact stats and controls**
- Replace 12-line stats panel with 2-row compact stats (Exported/Pending/Dead + Projects/KF)
- Inline overall progress indicator
- Merge controls into single row (Batch + Rescan + Start + Stop)
- Remove: Avg/chat, Avg/batch, ETA, Last stop, Last error, Delta

**US-004: Shrink log area**
- Reduce log textarea from 160px to ~80px
- Ensure batch summaries, errors, and scan results still log properly
- Task-level status moves to the task list (not duplicated in log)
