# Persistent Structured Logging for JS Exporter

> **Date**: 2026-03-16
> **Status**: Design Complete
> **Prerequisite**: Event-driven exporter architecture (ef7cfa3) — already implemented
> **Roadmap item**: "Logs for traceability" from `docs/ai/roadmap.md`

## Problem

The JS exporter runs for hours, sometimes overnight. When it breaks, there's no way to understand what happened:

- **Current logging is ephemeral**: `addLog(msg: string)` writes to a 200-line rolling buffer in `ExportState.logs[]`, displayed in an 80px textarea. Logs are **cleared on every startup** (`S.logs = []`).
- **No structured data**: Plain text strings with timestamps. No severity levels, no categories, no machine-parseable context.
- **`console.error`/`console.warn` uncaptured**: ~15 `console.error`/`console.warn` calls scattered across the codebase are invisible to the user — they go to the browser console which is closed during overnight runs.
- **No crash diagnostics**: Uncaught errors and unhandled promise rejections vanish silently.
- **No export**: The user cannot extract logs for offline analysis.

The user's stated goal: "full traceability... be able to detect when [the API] returns errors... details about the recoverability... understand what happened... put it all together as one journey."

## Design

### Architecture

One unified structured log system with two access patterns:

1. **Real-time monitoring** — the UI textarea shows the current session's logs (info+ level), formatted as human-readable text.
2. **Post-mortem analysis** — the full log history is persisted in a dedicated IDB database and exportable as JSONL.

```
Any component
  → log(level, category, message, context?)
  → In-memory array (current session, powers UI textarea)
  → IDB write (persistent, powers JSONL export)
```

The `log()` function is synchronous from the caller's perspective. IDB writes happen asynchronously in the background. If an IDB write fails, the entry still exists in the in-memory array for the current session.

### Log Entry Schema

```typescript
interface LogEntry {
  id: number;                         // Auto-increment PK (ordering key)
  timestamp: number;                  // Unix ms — for time-range queries
  session: string;                    // Session ID (generated once at startup)
  level: "debug" | "info" | "warn" | "error";
  category: string;                   // Domain category (see below)
  message: string;                    // Human-readable description
  context?: Record<string, unknown>;  // Structured data — self-contained, no cross-referencing needed
}
```

Entries are self-contained: the `context` object should include everything needed to understand the entry without querying other stores. For example, a conversation export log includes `conversationId`, `title`, and `fileCount` — not just the ID.

**Session ID**: Generated once per page load (e.g., 8-char random hex). Distinguishes separate runs in the export. Multiple Start/Stop cycles within one page load share the same session — "run" boundaries are inferred from lifecycle log entries.

### Log Levels

| Level | Semantics | UI textarea | JSONL export |
|-------|-----------|:-----------:|:------------:|
| `debug` | Verbose internals (event emissions, individual IDB writes, API call details) | Hidden | Included |
| `info` | Normal operations (scan started, item exported, run completed) | Shown | Included |
| `warn` | Recoverable issues (retry triggered, 429 rate limit, token refresh, fallback used) | Shown | Included |
| `error` | Failures (item dead-lettered, unrecoverable errors, uncaught exceptions) | Shown | Included |

### Categories

Aligned with the user's mental model (matching the UI's three progress bars and the event-driven architecture layers):

| Category | Scope |
|----------|-------|
| `sys` | Lifecycle — startup, shutdown, version, session start, uncaught errors |
| `scan` | Conversation & project scanners — discovery, pagination, dedup decisions |
| `chat` | Chat export queue — conversation download, parsing |
| `file` | File/attachment download queue |
| `kf` | Knowledge file download queue |
| `net` | Network layer — auth, rate limiting, retries, backoff, request success after retry |
| `ui` | User interactions — button clicks, setting changes, filter changes |
| `state` | State persistence — save, load, reset, reconcile |

### Storage

**Dedicated IDB database: `cvz-log`**

Separate from `cvz-export`, `cvz-discovery`, and `cvz-export-blobs`. Rationale:
- Logs can be cleared without affecting export state or blobs
- Different retention policies
- No risk of log growth interfering with export operations
- Can be versioned independently

**Object store: `entries`**
- keyPath: `id`, autoIncrement: true
- Index on `session` — for session-scoped queries on export (optional optimization)
- No other indexes — retention cleanup uses the `id` cursor (monotonically increasing), time filtering happens client-side after reading

**Write strategy**: Individual IDB `put()` per entry. IDB transactions are async and non-blocking. At the expected throughput (1-10 entries/second during active export), individual writes are fine. If profiling reveals contention, micro-batching (flush every 500ms) can be added as an internal optimization without changing the `log()` interface.

**Fallback**: If IDB is unavailable (localStorage fallback mode), logs are kept in the in-memory array only. No attempt to persist logs to localStorage — it's too small (5MB) and the user already gets a warning banner about the fallback. Logs from the current session are still viewable in the UI and exportable, but won't survive page reloads.

### Retention

- **Max entries**: 100,000 (~30MB at ~300 bytes/entry average — negligible compared to the 800MB+ export blobs)
- **Cleanup trigger**: On startup only. If count > 100,000, delete oldest entries to bring count to 80,000.
- **No mid-session cleanup**: A single long session may exceed 100,000 entries. That's fine — IDB handles it, and the in-memory array only holds the current session's entries.
- **Reset behavior**: The existing "Reset" button clears export state + blobs. Logs are **not** cleared — they're diagnostic data you'd want even after resetting export state. A `__cvz_clearLogs()` debug alias is exposed on `window` for manual cleanup.

### Integration

**Replace `addLog` with `log`:**

Current: `addLog: (msg: string) => void`
New: `log: (level: LogLevel, category: string, message: string, context?: Record<string, unknown>) => void`

All ~40 callsites are updated. Every `deps` interface that currently has `addLog` gets `log` instead. The migration is mechanical — TypeScript types enforce correctness at every callsite.

**Mapping examples** (current → new):

| Current call | New call |
|---|---|
| `addLog("Start.")` | `log("info", "sys", "Start")` |
| `addLog("Auth 401, refreshing token...")` | `log("warn", "net", "Auth 401, refreshing token")` |
| `addLog("✓ " + title)` | `log("info", "chat", "Conversation exported", { conversationId: id, title })` |
| `addLog("HTTP 429. Sleeping " + ms + "...")` | `log("warn", "net", "Rate limited", { status: 429, retryAfter: waitMs, backoffCount })` |
| `addLog("✗ " + title + " (attempt " + n + "): " + error)` | `log("warn", "chat", "Export failed, retrying", { conversationId: id, title, attempt: n, error })` |
| `addLog("✗ Dead-lettered: " + title)` | `log("error", "chat", "Dead-lettered", { conversationId: id, title, error })` |
| `addLog("Chat concurrency set to " + n)` | `log("info", "ui", "Concurrency changed", { queue: "chat", value: n })` |
| `console.error(e)` in coordinator | `log("error", "sys", "Unhandled coordinator error", { error: String(e) })` |

**Replace `console.error`/`console.warn`:**

All ~15 `console.error`/`console.warn` calls become `log("error", ...)` or `log("warn", ...)` calls. No monkey-patching of `console`.

**Capture uncaught errors:**

```typescript
window.addEventListener("error", (e) => {
  log("error", "sys", "Uncaught error: " + e.message, {
    filename: e.filename,
    lineno: e.lineno,
    colno: e.colno,
  });
});

window.addEventListener("unhandledrejection", (e) => {
  log("error", "sys", "Unhandled promise rejection", {
    reason: String(e.reason),
  });
});
```

**Startup log entry:**

On session start, emit a single `sys`/`info` entry with:
- Version string (already available as `VER`)
- Session ID
- Storage backend (IDB vs localStorage)
- User agent
- Timestamp (implicit in the entry)

This ensures every log export has the "who ran this and when" context.

### Export

**Format: JSONL (JSON Lines)**

One JSON object per line. Easy to parse programmatically (`jq`, Python, any language), easy to `grep`, streamable, no need to parse the whole file.

**Trigger**: "Download Logs" button in the UI panel. Reads all entries from the `cvz-log` IDB store, serializes each as a JSON line, creates a Blob, triggers a browser download.

**Filename**: `cvz-logs-{YYYY-MM-DD-HHmmss}.jsonl`

**Scope**: Exports all stored logs across all sessions. The `session` field in each entry allows post-hoc filtering by session. No in-app filtering UI — YAGNI. The user downloads and filters with external tools.

### UI Changes

Minimal changes to the floating panel:

1. **Textarea stays** — feeds from the in-memory log array instead of `S.logs`. Formatted as:
   ```
   [HH:MM:SS] [WARN/net] Rate limited (429), retry in 30s
   [HH:MM:SS] [INFO/chat] Exported: "My Chat" (3 files)
   ```
   The level+category prefix distinguishes entries visually. The `context` object is **not** shown in the textarea — that's what the JSONL export is for.

2. **Display filter**: The textarea shows `info`, `warn`, and `error` entries. `debug` is excluded to keep the real-time view clean. Debug entries are only available in the JSONL export.

3. **"Download Logs" button** — added next to the existing "Export State" button. Shows entry count: `Logs (1,234)`.

4. **Remove `S.logs`** from `ExportState` — logs no longer live in the export state. The `logs: string[]` field is removed from the type. Existing state files with the field are unaffected (extra fields are ignored on load). The state migration module handles removal cleanly.

5. **No color coding** — the textarea is a plain `<textarea>`, no rich text. Color coding would require switching to a styled `<div>`, which adds complexity for marginal value. YAGNI.

6. **No interactive filtering** — no level dropdown, no category checkboxes. The real-time view is for monitoring. Analysis happens offline with the JSONL export.

### Traceability: Reconstructing the Journey

The user's key concern: "put it all together as one journey to find and understand the behavior."

Entity IDs in the `context` object are the correlation keys:

- **Conversation lifecycle**: filter JSONL by `context.conversationId` → see discovery, queueing, export start, file discovery, completion/failure, all retries
- **File lifecycle**: filter by `context.fileId` → see queueing, metadata fetch, download, completion/failure
- **API retry sequence**: filter by `context.url` → see initial request, 429/5xx, backoff duration, retry attempts, eventual success/failure
- **Session overview**: filter by `session` → see everything that happened in one page load

Example retry trace in JSONL:
```json
{"timestamp":1710547200000,"session":"a1b2c3d4","level":"warn","category":"net","message":"Rate limited","context":{"url":"/backend-api/conversation/abc-123","status":429,"retryAfter":30000,"backoffCount":1}}
{"timestamp":1710547230000,"session":"a1b2c3d4","level":"info","category":"net","message":"Retry succeeded","context":{"url":"/backend-api/conversation/abc-123","status":200,"attempt":2}}
```

### Debug Alias

```typescript
(window as any).__cvz_clearLogs = async () => {
  await logStore.clear();
  console.log("convoviz: logs cleared");
};
```

### Testing

- **LogStore unit tests**: Write, read, count, cleanup operations against `fake-indexeddb` (already a pattern in the codebase's IDB tests).
- **`log()` function tests**: Verify entries are pushed to in-memory array with correct schema, and that IDB write is called.
- **JSONL export tests**: Verify serialization format — one JSON object per line, all fields present.
- **Integration**: Update existing tests that mock `addLog` to mock the new `log` function with matching signature.

### What This Does NOT Include

- **Log upload/import** — the roadmap mentioned "download and upload" but settled on "just download." Upload is YAGNI.
- **In-app log viewer with search/filter** — YAGNI. The textarea is for real-time monitoring; analysis happens offline.
- **Log shipping to external services** — out of scope. This is a browser bookmarklet, not a server.
- **Per-run log separation** — sessions and lifecycle events (Start/Stop) provide enough structure to identify runs. A separate `runId` field adds complexity without value.
