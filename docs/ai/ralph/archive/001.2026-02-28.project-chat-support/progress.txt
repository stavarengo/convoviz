## Codebase Patterns

- `js/script.js` is a standalone browser bookmarklet -- no Node.js, no test framework, no build system. Syntax-check with `node --check` after stripping the `javascript: ` prefix: `sed '1s/^javascript: //' js/script.js > /tmp/check.js && node --check /tmp/check.js`.
- State merge logic is in `mergeState()`. New top-level fields need a guard in `mergeState()` if they aren't a simple primitive (e.g., `if (!Array.isArray(out.projects)) out.projects = [];`).
- Cursor-based pagination (projects, project conversations) uses `cursor` field in response; `null` means last page. Offset-based pagination (regular conversations) uses `offset`/`total`.
- The `onPage` callback is the shared mechanism for adding items to `S.progress.pending` with deduplication via `pendingSet`.
- Graceful degradation pattern: inner try/catch that rethrows AbortError but logs+continues for all other errors.
- New state arrays in `progress` need both a `mergeState()` guard (`if (!Array.isArray(...))`) and, if they use a fail-count map, an explicit object spread merge.
- Batch export functions follow a standard structure: batchItems from pending, queue+workers, success/fail tracking, updatePendingAfterBatch cleanup, ZIP build+download, stats update.

## 2026-02-28 - US-001

- Added `scanProjects(signal, onProject)` function: paginates through `/backend-api/gizmos/snorlax/sidebar?owned_only=true&conversations_per_gizmo=0` using cursor-based pagination, extracts project definitions with shape `{gizmoId, name, emoji, theme, instructions, memoryEnabled, memoryScope, files: [{fileId, name, type, size}], raw}`
- Added `scanProjectConversations(gizmoId, signal, onPage)` function: paginates through `/backend-api/gizmos/{gizmoId}/conversations?cursor=0` using cursor-based pagination, normalizes each item to `{id, title, update_time, gizmo_id}`
- Updated `defaultState()`: added `projects: []` top-level field, `scan.totalProjects: 0`
- Updated `mergeState()`: added guard for `projects` array
- Updated `rescan()`: orchestrates regular scan -> project list scan -> per-project conversation scan; shares `onPage` callback and `pendingSet` for deduplication
- Project conversations stored in `S.progress.pending` with `gizmo_id` field, regular items have no `gizmo_id`
- `S.scan.total` includes both regular and project conversation counts
- `S.scan.snapshot` includes all conversation IDs from both sources
- Status text: "Scanning conversations... offset N" -> "Scanning projects... (page N)" -> "Scanning project chats: Name (X/Y)"
- Sidebar endpoint failure logs warning, continues with regular-only export
- Per-project conversation scan failure skips that project, continues with remaining
- Files changed: `js/script.js`
- **Learnings for future iterations:**

  - The project sidebar response has nested structure: `items[].resource.gizmo` for the gizmo definition, `items[].resource.context.files` for knowledge files. Access patterns need defensive chaining since the exact nesting may vary.
  - Project conversation endpoint uses cursor starting at `"0"` (string), not `null`. First page uses `cursor=0`, subsequent pages use the cursor from the response.
  - AbortError must always be rethrown in catch blocks during scan phases -- swallowing it would prevent the stop button from working.

## 2026-02-28 - US-003

- Added state fields to `defaultState()`: `progress.kfExported`, `progress.kfPending`, `progress.kfDead`, `progress.kfFailCounts` and `stats.kfBatches`, `stats.kfMs`, `stats.kfFiles`
- Updated `mergeState()`: explicit merge for `kfFailCounts` (same pattern as `failCounts`), plus array guards for `kfExported`, `kfPending`, `kfDead`
- Updated `rescan()`: after scanning projects, rebuilds `kfPending` from `S.projects` file lists, skipping files already in `kfExported` or `kfDead`
- Added `exportKnowledgeBatch(signal)` function mirroring `exportOneBatch()` patterns: batch/conc/pause settings, concurrent workers, retry with 3-attempt dead-letter
- Download URL: `GET /backend-api/files/download/{fileId}?gizmo_id={projectId}&inline=false`
- `file_not_found` errors are dead-lettered immediately (no retry), other HTTP errors follow normal 3-retry pattern
- ZIP structure: `projects/{projectId}/files/{fileId}_{sanitizedName}` for files, `projects/{projectId}/project.json` for project definitions
- `convoviz_export_meta.json` included in each KF ZIP with batch stats
- ZIP filename pattern: `convoviz_knowledge_YYYYMMDD_HHMMSS_nN.zip`
- Updated `Exporter.start()` main loop: scan -> conversation batches -> knowledge file batches -> done
- Stop mid-knowledge-download preserves `kfPending` for resume; conversations are checked first on restart
- Files changed: `js/script.js`
- **Learnings for future iterations:**

  - The `kfExported` array stores full item objects (not just IDs like conversation `exported`), because the rescan rebuild needs to match on `fileId`. This is a deliberate divergence from the conversation pattern.
  - When adding a new batch export function, the `file_not_found` immediate dead-letter must also mark `kfFailCounts[fileId] = 3` to prevent `updateKfPendingAfterBatch` from double-counting the item.
