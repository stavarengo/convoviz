# Skip Project Knowledge Files in Per-Conversation Export

## Problem

The bookmarklet's `exportOneBatch` downloads every file referenced in a conversation. ChatGPT Projects inject **project knowledge files** as attachments into every conversation within that project. This creates two problems:

1. **Redundant downloads that always fail.** Each project knowledge file is attempted via `/backend-api/files/download/{id}` (no `?gizmo_id=` param), which returns HTTP 500 for project-scoped files. For a project with 4 knowledge files and 45 conversations, this produces ~180 failed requests, each followed by a 300ms pause — adding ~54 seconds of pure waste.

2. **Malformed `file-service://` asset pointers.** `extractFileRefs` strips `sediment://` from `asset_pointer` fields but doesn't handle `file-service://`, causing the full URI to be sent as a file ID, which returns HTTP 422. This is a bug in the extraction logic.

Both issues are distinct from legitimate file failures (404s for deleted files, transient 500s for real conversation files) which should continue to be attempted.

## Evidence From Logs

| Pattern | Example | Root Cause | Count (45-conv batch) |
|---------|---------|------------|-----------------------|
| HTTP 500 same 4 IDs across all conversations | `file-93KbkaWiDMdKKaRY6whjL7` (Personal Linux Setup.md) | Project knowledge file, wrong endpoint | ~160 |
| "No download_url" for handful of IDs | `file-6w5tmTMqxso331Mx2z67Rq` | Likely old/removed project files (appear across many convos but not in current `S.projects`) | ~15 |
| HTTP 422 "Invalid file_id" | `file-service://file-NK928nPrA5ZpuYEUQga26F` | Prefix not stripped | 2 |
| HTTP 404 "File not found" | `file_000000003e6471f48bc6f6462a624061` | Deleted file on ChatGPT servers | 3 |

The 404s and "No download_url" for historical file IDs are **not addressed** by this change. The 404s are legitimate attempts for files that no longer exist. The "No download_url" files appear to be old/removed project files not present in the current `S.projects` state — there's no reliable way to identify them, and they account for only ~15 failures vs ~160 for current project files.

## Design

### Fix A: Filter project knowledge files from per-conversation download

**Where:** `exportOneBatch()`, after calling `extractFileRefs()` (line ~1368).

**How:** Build a `Set` of all project knowledge file IDs from `S.projects` once before the worker loop. After extracting refs, filter out any whose ID is in the set.

```js
// Before the worker loop (around line 1347):
const projectFileIds = new Set(
  (S.projects || []).flatMap(function(p) {
    return (p.files || []).map(function(f) { return f.fileId; });
  })
);

// At line 1368, replace:
//   const refs = extractFileRefs(detail);
// with:
const allRefs = extractFileRefs(detail);
const refs = allRefs.filter(function(f) { return !projectFileIds.has(f.id); });
```

**Why filter at the call site, not inside `extractFileRefs`:**
- `extractFileRefs` is a pure function (takes JSON, returns refs). It has no knowledge of `S.projects` and shouldn't.
- Filtering at the call site keeps the extraction logic reusable and testable.

**Why filter against ALL projects, not just the conversation's project:**
- File IDs are globally unique. A knowledge file in any project should never be downloaded via the per-conversation endpoint.
- Simpler — no need to resolve the conversation's project and handle the case where `item.gizmo_id` is null.

### Fix B: Handle `file-service://` prefix in `extractFileRefs`

**Where:** `extractFileRefs()`, line 902.

**How:** Replace the single `.replace("sediment://", "")` with a regex that strips both known prefixes.

```js
// Replace:
//   const id = String(p.asset_pointer).replace("sediment://", "");
// with:
const id = String(p.asset_pointer).replace(/^(?:sediment|file-service):\/\//, "");
```

This is intentionally narrow (explicit prefix list) rather than a generic `/.+:\/\//` strip, to avoid accidentally mangling unexpected formats.

## Impact Analysis

### What changes

| Aspect | Before | After |
|--------|--------|-------|
| File download attempts per conversation | All refs (including project files) | Only conversation-specific files |
| `files N` count in success log | Includes project files (inflated) | Only real conversation files (accurate) |
| `filesSaved` / `filesFailed` counters | Inflated by project file failures | Accurate |
| Batch summary `Files +N/-M` | Large `-M` from project file failures | Reflects actual failures |
| `file-service://` asset pointers | HTTP 422 | Proper file ID, may succeed |

### What does NOT change

- **Conversation export success/failure** — `successes.push(detail)` happens regardless of individual file download results. Filtering project files does not affect whether a conversation is marked as exported.
- **Knowledge file export** — The KF pipeline (`exportKnowledgeBatch`) is a completely separate code path that correctly uses `?gizmo_id=` and is untouched by this change.
- **ZIP format** — Project files were never successfully downloaded per-conversation anyway, so no data disappears from the output.
- **Non-project conversations** — `projectFileIds` would simply not match any of their file refs; all refs pass through the filter unchanged.
- **The `extractFileRefs` function** — It remains pure; only its output is filtered at the call site.

### Edge cases

| Scenario | Behavior |
|----------|----------|
| `S.projects` empty/not loaded | `projectFileIds` is empty Set, filter is no-op, falls back to current behavior |
| Project with 0 files | No file IDs added to Set, no filtering for that project |
| Conversation has both project files and real uploads | Project files filtered, real uploads attempted normally |
| Unknown `asset_pointer` prefix (e.g. `newprotocol://`) | Not stripped by the regex; full URI passes through as before — no regression |

## Test Considerations

The existing Playwright test "exportOneBatch sets task detail during fetch and download phases" (line 141) uses conversations with `attachments: [{id: "file-001"}, {id: "file-002"}]` and `S.projects` is not set. Since `projectFileIds` would be empty, **all existing tests pass unchanged**.

New test coverage needed:
- **Conversation with project file refs filtered:** Set `S.projects` with files, create a conversation whose attachments include those file IDs. Verify they are not downloaded. Verify the conversation still exports successfully.
- **`file-service://` prefix stripped:** Add an `asset_pointer` with `file-service://file-XXX` to a conversation. Verify the download is attempted for `file-XXX` (not the full URI).
