# Event-Driven Exporter Architecture

> **STATUS: WORK IN PROGRESS** — This document captures the user's vision for a full architectural rewrite of the JS exporter. It is the long-term goal, building on top of Plan 1 ("Decouple File Downloads"). A future session should continue this brainstorm from where it left off.

## Context

This plan was discussed in the same brainstorming session as Plan 1 (see `2026-03-15.000.decouple-file-downloads.md`). The user described a comprehensive event-driven, separation-of-concerns architecture. Plan 1 delivers the immediate value (decoupling file downloads via generic queues). This plan captures the broader vision for future implementation.

## Prerequisite

Plan 1 must be completed first. The generic queue abstraction from Plan 1 becomes the foundation for this architecture.

## User's Vision (Verbatim Summary)

The user envisions a fully event-driven, decoupled system where:

### Core Principles

1. **No component knows about queues directly.** Queues are registered during app initialization (bootstrap). The main process, scanners, and workers emit domain events — they don't reference queues.
2. **Domain-aligned events, not generic ones.** Events should be named with domain language (e.g., "conversation-needs-export", "project-discovered") rather than generic names (e.g., "item-found"). Event payloads should be minimal DTOs — just enough for the listener to know what to fetch from the source of truth (the database), not full data dumps.
3. **Queues are fully isolated and decoupled.** They listen to events, process work, and emit their own events. They don't know about each other.
4. **Everything is persistent and resumable.** Scanner pagination state is saved so it can resume after crashes. Records are inserted/updated in the database as items are discovered, before any processing.
5. **Smart deduplication at every level.** When a conversation ID is discovered, the system checks the database: is this new? Does it need updating (compare update_time)? Is it already up-to-date? Different events are emitted for each case.

### Discovery Layer

#### Conversation Scanner

- A reusable, generic component that paginates the ChatGPT conversations API
- Can scan general conversations OR project-specific conversations (same code, different parameters — project-specific just adds the `gizmo_id` filter)
- For each conversation discovered:
  1. Check database for existing record
  2. Compare `update_time` to determine if refresh is needed
  3. Insert or update the record
  4. Emit a domain event based on what happened:
     - `conversation-needs-export` — new conversation, never processed
     - `conversation-needs-update` — existing conversation, `update_time` changed
     - `conversation-up-to-date` — no action needed (informational event only, no queue listens)
- Persists its own pagination state (offset, limit, total) so it can resume from where it stopped
- The algorithm should NOT blindly trust saved pagination state — it should always handle the case where a record is already in the database (since the API uses offset/limit pagination, not cursors, duplicates will occur)
- Multiple instances can run in parallel (one for general conversations, one per project)

#### Project Scanner

- Paginates the project/gizmo sidebar API
- For each project discovered:
  1. Insert/update project record in database
  2. Emit `project-discovered` event
- Two listeners react to `project-discovered`:
  1. **Conversation Scanner listener**: creates a new Conversation Scanner instance filtered to that project's `gizmo_id` (same reusable scanner, different parameter)
  2. **Knowledge File listener**: starts processing knowledge files for that project

### Processing Layer (Event → Queue)

#### Conversation Exporter

- Listens to: `conversation-needs-export`, `conversation-needs-update`
- Worker:
  1. Fetches full conversation JSON from `/backend-api/conversation/{id}`
  2. Parses and stores the conversation data
  3. During parsing, discovers file references
  4. Emits `conversation-exported` or `conversation-files-discovered` (or both — the design should emit focused, specific events)
- The conversation exporter does NOT know about the file download queue. It just emits events about what it found.

#### File Downloader

- Listens to: `conversation-files-discovered` (from conversation exporter) AND knowledge file discovery events (from project scanner)
- Both conversation attachments and knowledge files enter the same download queue
- Each file item carries a `type` flag: `"attachment"` vs `"knowledge-file"` — used during ZIP generation to determine folder placement
- Worker:
  1. Check if file already downloaded (dedup)
  2. Fetch metadata
  3. Download blob
  4. Store to IDB
- The file downloader does NOT know where file references come from — it just processes items from its queue

### Knowledge Files

The user's key insight: once knowledge files are discovered (from project metadata), their download process is identical to conversation attachments. The difference is only:
- **Discovery source**: project metadata vs. conversation JSON parsing
- **API call**: requires `gizmo_id` parameter
- **ZIP placement**: `kf/{projectName}/` folder vs. root-level files

In Plan 2, these could be unified into a single file download queue with a type flag, rather than separate queues.

### Event Flow Diagram

```
App Bootstrap
  ├── Register event listeners
  ├── Initialize queues via generic queue abstraction
  └── Start scanners

Conversation Scanner (general)  ─── emits ──→  conversation-needs-export
                                               conversation-needs-update
                                               conversation-up-to-date

Project Scanner ────────────── emits ──→  project-discovered
                                              │
                               ┌──────────────┼──────────────┐
                               ▼              ▼              ▼
                    Conversation Scanner   Knowledge File   (other future
                    (per project)          Scanner           listeners)
                          │                    │
                          ▼                    ▼
                    conversation-needs-export  knowledge-file-discovered
                          │                    │
                          ▼                    ▼
                    Conversation Exporter   File Downloader Queue
                          │                  (unified for all file types)
                          ▼
                    conversation-exported
                    conversation-files-discovered
                          │
                          ▼
                    File Downloader Queue
```

### Event Implementation

- Events can use browser-native APIs if appropriate, or an internal pub/sub mechanism
- Event payloads should be minimal DTOs (just IDs and metadata needed to fetch from the source of truth)
- The user emphasized: "Nobody would know about the queues. It's just that we set up the queues during app initialization, and the abstract queue mechanism is going to have these listeners."

### Data Persistence

- All discovery results (conversation IDs + metadata, project IDs) are persisted to the database (IDB) immediately upon discovery, BEFORE emitting events
- This ensures crash resilience — even if the app closes, the discovered items are already in the database
- The event is about signaling that processing should happen, not about carrying the data itself

### Database Design Considerations

The user mentioned:
- Possibly separate tables/stores for conversations vs. projects
- Or the same table with different fields
- Left the exact schema design to the implementer
- Key requirement: each record type needs enough metadata to determine if it needs re-processing

### Relationship to Plan 1

Plan 1's generic queue abstraction is the foundation. Plan 2 adds:
1. **Event bus**: pub/sub system for domain events
2. **Scanner abstraction**: reusable, resumable pagination component
3. **Listener registration**: connecting domain events to queues during bootstrap
4. **Smart deduplication**: database checks before emitting events
5. **Unified file download**: single queue for both attachments and knowledge files

The queue abstraction, IDB storage, UI components, and rate limiting from Plan 1 are reused directly.

## Open Design Questions (for next session)

1. **Event bus implementation**: Browser-native (CustomEvent on a shared target) vs. in-memory pub/sub class? The bookmarklet context may evolve (user said "it's a bookmarklet now, but let's not get limited by this").
2. **Scanner state persistence**: How to persist pagination cursor/offset for resumability? Separate IDB store or fields in ExportState?
3. **Dedup granularity**: Should dedup happen at event emission (scanner checks DB before emitting) or at the listener/worker level (worker checks DB before processing)? User seemed to prefer both: scanner emits specific events (needs-export vs. up-to-date), and worker also validates before processing.
4. **Conversation Scanner reusability**: How to parameterize for general vs. project-specific without if/else branching? Different API endpoints (`/backend-api/conversations` vs. `/backend-api/gizmos/{id}/conversations`) but similar response shapes.
5. **Knowledge file unification**: Can knowledge file downloads truly use the same queue as conversation attachments? The API call is different (requires `gizmo_id`). A type flag + conditional logic in the worker might work, or the worker could delegate to a type-specific handler.
6. **Scope boundary**: At what point does this become "build a framework"? The user explicitly wanted the queue system to be generic and reusable. But the event bus, scanner abstraction, and listener registration add significant framework-level code. Where's the line?

## Technical Notes

- The user confirmed this can run as a browser console script, not limited to bookmarklet constraints. The architecture can be more sophisticated than what a bookmarklet would typically support.
- The current codebase already has patterns that support this direction: dependency injection via `deps` objects, modular file structure, separate scan/export/storage layers.
- Current IDB usage: `cvz-export` for state, `cvz-export-blobs` for conversation JSON + file blobs. Plan 2 may need additional IDB stores for scanner state and the event-driven persistence model.
