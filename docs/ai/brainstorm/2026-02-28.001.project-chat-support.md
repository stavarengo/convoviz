# Bookmarklet: Project & Knowledge File Support

## Context

The Convoviz bookmarklet (`js/script.js`, v2.0) exports ChatGPT conversations and their media. After the reliability overhaul (IndexedDB, unified fetch, circuit breaker), it is stable and production-ready for **regular conversations** and **custom GPT conversations**.

However, it completely misses **ChatGPT Projects** — a feature where users organize conversations into project workspaces with dedicated knowledge files, instructions, and scoped memory.

### What We Learned from the API (HAR Analysis)

**Projects are gizmos with `g-p-` prefix.** Internally, ChatGPT treats projects as a variant of the "gizmo" entity (same as custom GPTs), differentiated by `gizmo_type: "snorlax"` and ID prefix `g-p-` (vs `g-` for custom GPTs).

**The two conversation lists are disjoint:**

| Endpoint | Pagination | Returns |
|---|---|---|
| `GET /backend-api/conversations?offset=N&limit=N&order=updated` | offset/limit/total | Regular + custom GPT chats. **Not** project chats (with rare exceptions). |
| `GET /backend-api/gizmos/{g-p-id}/conversations?cursor=0` | cursor (null = done) | Only conversations within that project. |

Verified empirically: 168 regular conversations and 80 project conversations with **zero overlap** across endpoints.

**Project listing uses the sidebar endpoint:**

| Endpoint | Pagination | Returns |
|---|---|---|
| `GET /backend-api/gizmos/snorlax/sidebar?owned_only=true&conversations_per_gizmo=0` | cursor (null = done) | All user's projects, each with gizmo definition + knowledge files list. |

**Knowledge files** are attached to projects (not to individual conversations). Each project's gizmo definition includes a `files` array with `{id, file_id, name, type, size}`. Downloaded via `GET /backend-api/files/download/{file_id}?gizmo_id={g-p-id}`.

**Conversation detail is identical** for all types. `GET /backend-api/conversation/{uuid}` returns the same structure for regular, custom GPT, and project conversations. The existing export pipeline handles them all without changes.

## Scope

This design covers:
- Scanning projects and their conversations alongside regular conversations
- Exporting project conversations through the existing pipeline (no changes to conversation export logic)
- Downloading project knowledge files (new phase, same patterns as conversation export)
- Minimal UI additions to surface project/knowledge-file progress
- Project metadata in export ZIPs

This design does **not** cover:
- UI overhaul with per-task progress (separate PRD-2, informed by the whiteboard sketch)
- Custom GPT definition export (conversations already exported, definitions are a different feature)
- Stale export detection (re-exporting updated conversations)
- Shared/team projects (`owned_only=true` only)
- Project file editing/management

## 1. Scan Phase

### Current Flow

`Exporter.rescan()` (line 753) orchestrates the scan:
1. Builds `exportedSet`, `deadSet`, `pendingSet` from persisted state
2. Calls `scanConversations(signal, onPage)` — paginates through `/backend-api/conversations` (offset-based)
3. `onPage` callback adds new items to `S.progress.pending` incrementally (UI updates in real-time)
4. After scan: computes changes, updates `scan.snapshot`, saves state

### New Flow

After regular conversation scanning, two new phases:

**Phase 2 — Scan project list:**
- New function: `scanProjects(signal, onProject)`
- Paginates through `GET /backend-api/gizmos/snorlax/sidebar?owned_only=true&conversations_per_gizmo=0`
- Cursor-based: append `&cursor={value}` for subsequent pages, stop when `cursor` is `null`
- For each project, extract and store the gizmo definition (including files list) in `S.projects`
- Calls `onProject(projectDef)` per item for incremental UI feedback
- Returns array of all project definitions

**Phase 3 — Scan project conversations:**
- For each project from Phase 2, call new function: `scanProjectConversations(gizmoId, signal, onPage)`
- Paginates through `GET /backend-api/gizmos/{gizmoId}/conversations?cursor=0`
- Cursor-based: same pattern as sidebar pagination
- Each conversation item is normalized to `{id, title, update_time, gizmo_id}` — same shape as regular items, plus `gizmo_id`
- Uses the same `onPage` callback to incrementally add to `S.progress.pending`
- The existing `pendingSet` deduplication prevents the rare edge case of a project chat appearing in both endpoints

**After all three phases:**
- `scan.total` = regularCount + allProjectConversationsCount
- `scan.snapshot` includes all conversation IDs from both sources (same `[[id, update_time], ...]` format)
- `S.projects` contains all project definitions (for knowledge file tracking)
- Knowledge file pending queue is rebuilt from project file lists (see Section 4)

### Status Text During Scan

```
"Scanning conversations... offset 150"        ← Phase 1 (existing)
"Scanning projects... (page 2)"               ← Phase 2 (new)
"Scanning project chats: App Checker (3/21)"  ← Phase 3 (new)
```

### Error Handling

- **Sidebar endpoint fails**: Log warning `"⚠ Could not scan projects — continuing with regular chats only"`. Continue with regular conversations. No hard failure.
- **Per-project conversation scan fails**: Log error for that project, skip it, continue with remaining projects.
- **Token expiry during project scan**: Handled by existing `_fetch()` 401 retry + token refresh.

### Conversation Item Shape


The `gizmo_id` field is `null` for regular and custom GPT conversations, `"g-p-..."` for project conversations. Old v1 items without `gizmo_id` default to `undefined` which works equivalently to `null` — no migration needed.

### Knowledge File Item Shape

```js

```

### Backward Compatibility

`mergeState()` (line 228) already spreads `{...defaultState(), ...savedState}` for each nested object. New fields from `defaultState()` are applied automatically when loading old v1 state. The only addition needed: merge `kfFailCounts` explicitly (same pattern as existing `failCounts` merge on line 243-245).

### Project Definition Shape (stored in `S.projects`)

```js
{
  gizmoId: "g-p-69955c9b...",
  name: "App Checker",
  emoji: "kettlebell",                  // display icon (null if unset)
  theme: "#FF66AD",                     // color theme (null if unset)
  instructions: "...",                  // system prompt
  memoryEnabled: true,
  memoryScope: "project_v2",
  files: [                              // knowledge files attached to project
    { fileId: "file-325wxvgc8...", name: "guide.pdf", type: "application/pdf", size: 12345 }
  ],
  raw: { ... }                          // full gizmo API response (for project.json in ZIP)
}
```

The `raw` field stores the full sidebar response for this project (`{gizmo: {...}, tools: [], files: [...], product_features: {...}}`). This is included verbatim as `project.json` in knowledge file ZIPs — preserving all data for the Python side without assuming what it needs.

## 3. Conversation Export

**No changes to the core export logic.** Project conversations flow through the existing `exportOneBatch()` pipeline identically to regular conversations:
- Fetched via `GET /backend-api/conversation/{id}` (works for all types)
- Message-level attachments extracted by `extractFileRefs()` (same JSON structure)
- Files downloaded via existing attachment download logic
- Added to `conversations.json` in the batch ZIP

### Minor Enhancements

**Status text context:** When processing a pending item with `gizmo_id`, look up the project name from `S.projects` and display:
```
"Fetching: [App Checker] Userscript Exporter Analysis"
```
instead of just:
```
"Fetching: Userscript Exporter Analysis"
```

**Export metadata:** The `convoviz_export_meta.json` in each batch ZIP gains a `projects` field containing all project definitions (minus `raw` and `instructions` to keep it lightweight). This lets the Python side associate any conversation's `conversation_template_id` to a project name/emoji without needing separate files.

## 4. Knowledge File Download

A new export phase that runs **after all conversation batches are done**. Follows the same batch/retry/dead-letter patterns as conversation export for consistency.

### New Function: `exportKnowledgeBatch(signal)`

Mirrors `exportOneBatch()`:

1. Take a slice of `S.progress.kfPending` (size = `S.settings.batch`)
2. Create a `ZipLite` instance
3. For each file (concurrent workers, same as conversations):
   a. `GET /backend-api/files/download/{fileId}?gizmo_id={projectId}&inline=false`
   b. If response `status === "success"` → fetch blob from `download_url`
   c. If response `status === "error"` (e.g., `file_not_found`) → dead-letter immediately, no retry
   d. If HTTP error → normal retry (3 attempts then dead-letter)
   e. Add blob to ZIP at `projects/{projectId}/files/{fileId}_{sanitizedName}`
4. For each project that has files in this batch, add `projects/{projectId}/project.json` (the full `raw` gizmo definition)
5. Add `convoviz_export_meta.json` with batch stats
6. Build ZIP, trigger download with filename: `convoviz_knowledge_YYYYMMDD_HHMMSS_nN.zip`
7. Update `kfExported`, `kfPending`, handle failures → `kfDead`

### Integration into `Exporter.start()`

The main loop becomes:

```
1. Scan (if needed)                     ← existing + project scanning
2. While pending conversations → exportOneBatch()    ← existing
3. While kfPending → exportKnowledgeBatch()          ← NEW
4. Done
```

If the user stops mid-knowledge-download, `kfPending` persists. On next Start, conversations are checked first (any new pending?), then knowledge files resume.

### Knowledge File Pending Queue Rebuild

During `rescan()`, after scanning projects:
- Build `kfExportedSet` from `S.progress.kfExported`
- Build `kfDeadSet` from `S.progress.kfDead`
- For each project in `S.projects`, for each file in `project.files`:
  - If `fileId` not in `kfExportedSet` and not in `kfDeadSet` → add to `kfPending`
- This ensures new knowledge files (added since last scan) are picked up, and already-downloaded files are not re-downloaded

## 5. ZIP Structure

### Conversation Batch ZIP (minimal change)

```
conversations.json                              ← array of conversation details (unchanged)
{file_id}_{name}                                ← message-level attachments, flat (unchanged)
convoviz_export_meta.json                       ← batch meta + NEW projects array
```

### Knowledge File Batch ZIP (new)

```
projects/{gizmo_id}/project.json                ← full gizmo definition (raw API response)
projects/{gizmo_id}/files/{file_id}_{name}      ← knowledge files
convoviz_export_meta.json                       ← batch meta
```

### Design Decisions

- **Message attachments stay flat** in conversation ZIPs. The `projects/` subdirectory structure is exclusively for project-level knowledge files. This avoids changing the existing ZIP layout that the Python side already parses.
- **Each ZIP is self-contained.** Project metadata is repeated in every conversation batch's `convoviz_export_meta.json`. Redundant but harmless — the Python side processes ZIPs independently.
- **`project.json` is the raw API response.** No data loss. The Python side extracts what it needs.

## 7. Error Handling

| Scenario | Behavior |
|---|---|
| Sidebar endpoint fails (403, network) | Log warning, skip project scan entirely. Regular conversations still exported. |
| Per-project conversation scan fails | Log error, skip that project. Other projects still scanned. |
| Knowledge file `file_not_found` | Dead-letter immediately. File is gone from OpenAI storage — no retry. |
| Knowledge file HTTP error (5xx, network) | Normal retry (3 attempts). Dead-letter after 3 failures. |
| Knowledge file `download_url` fetch fails | Retry. CDN URLs can have transient failures. |
| Token expires during project scan | Handled by existing `_fetch()` 401 retry + token refresh. |
| User stops mid-knowledge-download | `kfPending` persists. Resume on next Start. |

**Graceful degradation is the principle:** no single failure blocks the entire export. Project scan failure degrades to regular-only export. Individual file failures are dead-lettered. The user always gets as much data as possible.

## 8. Suggested User Stories

**US-001: Scan and export project conversations**
- Scan project list via `/gizmos/snorlax/sidebar` (cursor-paginated)
- Scan conversations per project via `/gizmos/{id}/conversations` (cursor-paginated)
- Merge project conversations into the existing pending queue with `gizmo_id` field
- Store project definitions in `S.projects`
- Project conversations export through the existing `exportOneBatch()` pipeline (no changes to export logic)
- Add `projects` array to `convoviz_export_meta.json` in each batch ZIP
- Graceful degradation: sidebar failure → regular-only export

**US-002: Download and export project knowledge files**
- New state fields: `kfPending`, `kfExported`, `kfDead`, `kfFailCounts`
- Rebuild `kfPending` during each rescan from project file lists
- New `exportKnowledgeBatch()` function (mirrors `exportOneBatch()` patterns)
- Downloads via `/backend-api/files/download/{fileId}?gizmo_id={projectId}`
- `file_not_found` → immediate dead-letter
- ZIP structure: `projects/{gizmoId}/project.json` + `projects/{gizmoId}/files/{fileId}_{name}`
- Runs after all conversation batches complete

**US-003: UI updates for project support**
- Stats panel: "Projects: N" and "KF: X/Y" counters
- Secondary progress bar (purple) for knowledge file phase
- Enhanced status text with project context during scan and export
- Version bump to `cvz-bookmarklet-3.0`

## API Reference

Endpoints used (new ones marked with *):

| Endpoint | Method | Auth | Pagination | Purpose |
|---|---|---|---|---|
| `/backend-api/conversations?offset=N&limit=N&order=updated` | GET | Yes | offset/limit/total | Regular + custom GPT conversations |
| * `/backend-api/gizmos/snorlax/sidebar?owned_only=true&conversations_per_gizmo=0` | GET | Yes | cursor (null = done) | List all projects with metadata + files |
| * `/backend-api/gizmos/{g-p-id}/conversations?cursor=0` | GET | Yes | cursor (null = done) | Conversations within a project |
| `/backend-api/conversation/{uuid}` | GET | Yes | — | Full conversation detail (all types) |
| `/backend-api/files/download/{file_id}` | GET | Yes | — | File download URL (message attachments) |
| * `/backend-api/files/download/{file_id}?gizmo_id={g-p-id}` | GET | Yes | — | File download URL (knowledge files, gizmo_id for auth) |
| `/api/auth/session` | GET | Cookie | — | Access token refresh |
