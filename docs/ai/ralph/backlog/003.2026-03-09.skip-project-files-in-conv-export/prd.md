# PRD: Skip Project Knowledge Files in Per-Conversation Export

**Project:** Convoviz
**Branch:**

## Introduction

The convoviz bookmarklet exports ChatGPT conversations by downloading each conversation's JSON and its referenced files. ChatGPT Projects inject **project knowledge files** (e.g. `.zshrc`, `system-report.txt`) as attachments into every conversation within that project. The bookmarklet currently tries to download these project files via the per-conversation endpoint (`/backend-api/files/download/{id}` without `?gizmo_id=`), which always returns HTTP 500. This produces hundreds of failed requests and adds ~54 seconds of wasted time per 45-conversation batch.

Separately, `extractFileRefs()` only strips the `sediment://` prefix from `asset_pointer` fields but doesn't handle `file-service://`, causing full URIs to be sent as file IDs (HTTP 422).

Both issues are fixable with small, targeted changes in `js/script.js`.

## Goals

- Eliminate redundant HTTP 500 failures for project knowledge files during per-conversation export
- Fix `file-service://` prefix handling so those file IDs have a chance to download successfully
- Keep `extractFileRefs()` pure — filter at the call site, not inside the extraction function
- Ensure no data loss — project files are already correctly downloaded via the separate KF (Knowledge File) pipeline
- Maintain all existing test compatibility

## User Stories

### US-001: Filter project knowledge files from per-conversation downloads
**Status:** pending
**Description:** As a user exporting conversations, I want project knowledge files to be skipped during per-conversation file downloads so that the export completes faster without hundreds of spurious HTTP 500 errors.

**Acceptance Criteria:**
- [ ] In `exportOneBatch()`, before the worker loop (~line 1347), build a `Set<string>` of all project knowledge file IDs from `S.projects`:
  ```js
  const projectFileIds = new Set(
    (S.projects || []).flatMap(function(p) {
      return (p.files || []).map(function(f) { return f.fileId; });
    })
  );
  ```
- [ ] After calling `extractFileRefs(detail)` (~line 1368), filter out project file IDs before downloading:
  ```js
  const allRefs = extractFileRefs(detail);
  const refs = allRefs.filter(function(f) { return !projectFileIds.has(f.id); });
  ```
- [ ] When `S.projects` is empty or not loaded, `projectFileIds` is an empty Set and all refs pass through unchanged (no regression)
- [ ] Conversations with both project files and real user uploads: only user uploads are attempted, project files are skipped
- [ ] The conversation still exports successfully (`successes.push(detail)` still fires) regardless of filtering
- [ ] The `files N` count in the success log reflects only conversation-specific files (not project files)
- [ ] Add a Playwright test in `js/tests/export-pipeline.spec.js`: set `S.projects` with files, create a conversation whose attachments include those file IDs plus a regular file. Verify the project file IDs are NOT requested via `/backend-api/files/download/`, the regular file IS requested, and the conversation still exports as "done"
- [ ] Existing tests in `js/tests/export-pipeline.spec.js` continue to pass unchanged

### US-002: Strip `file-service://` prefix from asset pointers
**Status:** pending
**Description:** As a user exporting conversations with `file-service://` asset pointers, I want the prefix stripped so the file ID is valid and the download can succeed.

**Acceptance Criteria:**
- [ ] In `extractFileRefs()` (~line 902), replace the single `.replace("sediment://", "")` with a regex that strips both known prefixes:
  ```js
  const id = String(p.asset_pointer).replace(/^(?:sediment|file-service):\/\//, "");
  ```
- [ ] Unknown prefixes (e.g. `newprotocol://file-XXX`) are NOT stripped — current pass-through behavior is preserved
- [ ] Plain file IDs without any prefix (e.g. `file-XXX`) continue to work unchanged
- [ ] Add a Playwright test in `js/tests/export-pipeline.spec.js`: create a conversation with an `asset_pointer` of `file-service://file-test123`. Verify the download is attempted for file ID `file-test123` (not the full URI)
- [ ] Existing tests in `js/tests/export-pipeline.spec.js` continue to pass unchanged

## Functional Requirements

- FR-1: Build a `Set` of project knowledge file IDs from `S.projects` in `exportOneBatch()`, before the worker loop starts
- FR-2: After calling `extractFileRefs()`, filter the returned refs against the project file ID set before iterating for downloads
- FR-3: Update the `asset_pointer` prefix stripping in `extractFileRefs()` to handle both `sediment://` and `file-service://` using the regex `/^(?:sediment|file-service):\/\//`
- FR-4: Do not modify `extractFileRefs()` itself for project file filtering — keep it pure. Only filter at the call site in `exportOneBatch()`
- FR-5: The `js/script.min.js` file is auto-generated by the pre-commit hook (terser) — do not manually edit it

## Non-Goals

- Do NOT address HTTP 404 failures for deleted files (`file_000000...` IDs) — these are legitimate attempts for files that no longer exist on ChatGPT's servers
- Do NOT address "No download_url" failures for historical/removed project files not in current `S.projects` — there is no reliable way to identify them
- Do NOT modify the Knowledge File (KF) export pipeline (`exportKnowledgeBatch`) — it already works correctly with `?gizmo_id=`
- Do NOT use a generic regex like `/.+:\/\//` for prefix stripping — be explicit about known prefixes to avoid mangling unexpected formats
- Do NOT change the ZIP output format

## Technical Considerations

- The bookmarklet uses a mix of arrow functions and `function(){}` syntax. The code around `exportOneBatch` uses `function(){}` — new code should match this style for consistency.
- `Array.prototype.flatMap` is used in the proposed code. This is supported in all major browsers since ~2019 and is safe for the bookmarklet's target environment (modern browsers on chatgpt.com).
- `extractFileRefs()` is only called once (line 1368 in `exportOneBatch`). It is a pure function and is not used by the KF pipeline.
- The pre-commit hook in `.git/hooks/pre-commit` auto-minifies `js/script.js` → `js/script.min.js` using terser when `js/script.js` is staged.
- Existing Playwright tests in `js/tests/export-pipeline.spec.js` do not set `S.projects`, so the `projectFileIds` Set will be empty, and all refs will pass through the filter unchanged — no test breakage.

## Success Metrics

- Zero HTTP 500 errors for known project knowledge file IDs during per-conversation export
- Zero HTTP 422 errors for `file-service://` prefixed asset pointers
- All existing Playwright tests pass without modification
- Export batch time reduced (no 300ms pause per failed project file download)
