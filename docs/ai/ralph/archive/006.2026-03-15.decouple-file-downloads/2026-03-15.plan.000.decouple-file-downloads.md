# Decouple File Downloads from Chat Export

## Problem Statement

With ~4,000 chats, many containing multiple file attachments (some large), the JS exporter's export process is dominated by file download time. In `exporter.ts:630-701`, each conversation worker downloads ALL file attachments inline before marking the conversation as done. A single large file blocks one of the 3 available worker slots, preventing new conversations from being fetched.

## Goals

1. File downloads never block conversation export progress
2. Three concurrent, independent queues: conversations, attachments, knowledge files
3. Generic queue abstraction reusable for future queue types
4. Per-queue progress visibility in the UI
5. "Download what's ready" at any moment without stopping queues
6. Minimal changes to existing IDB storage model

## Non-goals (deferred to Plan 2: Event-Driven Exporter Architecture)

- Event-driven architecture with domain events
- Unified discovery layer with resumable scanners
- Smart deduplication at scan/discovery level
- Knowledge files unified into the file download flow
- Service class abstractions
- Conversation state tracking (new vs. needs-update vs. up-to-date)

## Architecture Overview

```
Current flow:
  Scan → exportOneBatch (fetch JSON + download files + store) → exportKnowledgeBatch

New flow:
  Scan → populates Chat Queue + Knowledge File Queue
         ┌─────────────────────────────────────────────────────┐
         │  Chat Queue (conc=3)                                │
         │  ├── fetch conversation JSON                        │
         │  ├── extract file refs → push to Attachment Queue   │
         │  ├── store JSON to IDB                              │
         │  └── mark exported                                  │
         │                                                     │
         │  Attachment Queue (conc=3, runs in parallel)         │
         │  ├── dedup check (skip if already in IDB)           │
         │  ├── fetch file metadata                            │
         │  ├── download blob                                  │
         │  └── store blob to IDB                              │
         │                                                     │
         │  Knowledge File Queue (conc=3, runs in parallel)    │
         │  ├── fetch file metadata                            │
         │  ├── download blob                                  │
         │  └── store blob to IDB at kf/{project}/{name}       │
         └─────────────────────────────────────────────────────┘

  User clicks "Download" at any time → ZIP from current IDB snapshot
```

All three queues share the same `Net` instance (same rate limiting/backoff). The scan phase is unchanged — it populates pending lists, then queues start. All three queues run concurrently from the start.

## Queue Abstraction (Generic)

The core new code. This is the reusable foundation for any queue type.

### Interface

```typescript
interface QueueItem {
  id: string;
}

interface QueueConfig<T extends QueueItem> {
  name: string;
  concurrency: number;
  maxRetries: number;
  pauseMs: number;
  worker: (item: T, signal: AbortSignal) => Promise<void>;
}

interface QueueCallbacks<T extends QueueItem> {
  onItemDone?: (item: T) => void;
  onItemFailed?: (item: T, error: string, attempt: number) => void;
  onItemDead?: (item: T, error: string) => void;
  onDrained?: () => void;
  onStatsChanged?: () => void;
}

interface QueueStats {
  pending: number;
  active: number;
  done: number;
  dead: number;
}

interface Queue<T extends QueueItem> {
  readonly name: string;
  readonly stats: QueueStats;
  readonly isRunning: boolean;
  enqueue(items: T[]): void;
  start(signal: AbortSignal): Promise<void>;
  stop(): void;
  setConcurrency(n: number): void;
}
```

### Key Behaviors

- **Workers are long-lived async loops**: each pulls one item from pending, processes it, pauses, repeats. When pending is empty, workers park (wait) using a Promise-based wake mechanism — no polling.
- **`enqueue()` wakes parked workers**: when new items arrive while workers are idle, they wake up immediately.
- **`start()` returns a Promise** that resolves when `stop()` is called or the queue drains (pending=0, active=0). The `onDrained` callback fires when the queue empties.
- **Retry/dead-letter is per-item**: on worker error, the queue increments the item's fail count. If below `maxRetries`, the item goes back to pending. If at `maxRetries`, the item goes to the dead list and `onItemDead` fires.
- **`setConcurrency()` while running**: increase spawns new workers. Decrease lets excess workers finish their current item then exit.
- **AbortError is special**: if the worker throws AbortError, the worker loop exits without counting it as a failure.
- **The `batch` concept is removed**: The current exporter processes items in batches, evaluating retries at batch boundaries. With the continuous queue model, this is handled per-item. Workers pull items one at a time. The batch size UI input is replaced by per-queue concurrency inputs.

### Worker Wake Mechanism

Workers park when the queue is empty and wake when new items arrive:

```typescript
// Conceptual implementation
let _wake: (() => void) | null = null;

enqueue(items: T[]) {
  this.pending.push(...items);
  if (_wake) { _wake(); _wake = null; }
}

async waitForItems(signal: AbortSignal): Promise<void> {
  if (this.pending.length) return;
  return new Promise((resolve, reject) => {
    _wake = resolve;
    signal.addEventListener('abort',
      () => reject(new DOMException('Aborted', 'AbortError')),
      { once: true }
    );
  });
}
```

## Queue Instances

### Chat Queue

- **Item type**: `PendingItem` (existing: `{ id, title, update_time, gizmo_id }`)
- **Worker**:
  1. Fetch conversation JSON: `net.fetchJson("/backend-api/conversation/" + item.id)`
  2. Extract file refs: `extractFileRefs(chatJson)`
  3. Push file refs to attachment queue: `attachmentQueue.enqueue(fileRefsAsAttachmentItems)`
  4. Store conversation JSON: `exportBlobStore.putConv(item.id, json)`
  5. Record export: `S.progress.exported[item.id] = item.update_time`
- **Concurrency**: Default 3, configurable via UI
- **Max retries**: 3
- **Initial items**: `S.progress.pending` (populated by scan)

### Attachment Queue

- **Item type** (new):
  ```typescript
  interface AttachmentItem extends QueueItem {
    id: string;           // fileId (unique identifier)
    name: string | null;  // original filename
    conversationId: string;
    conversationTitle: string;
  }
  ```
- **Worker**:
  1. **Dedup check**: query IDB for existing file key. If file already downloaded, skip (no-op success).
  2. Fetch file metadata: `net.fetchJson("/backend-api/files/download/" + item.id)`
  3. If no `download_url` in response: fail with descriptive error.
  4. Download blob: `net.fetchBlob(meta.download_url, { credentials based on origin })`
  5. Compute filename: `{fileId}_{sanitizedName}` or `{fileId}.{ext}` (same naming as current)
  6. Store blob: `exportBlobStore.putFile(fname, blob)`
- **Concurrency**: Default 3, configurable via UI
- **Max retries**: 3
- **Initial items**: empty — receives items dynamically from chat queue workers
- **Dedup handles page reloads**: on resume, the worker checks IDB before downloading, so already-downloaded files are skipped

### Knowledge File Queue

- **Item type**: `KnowledgeFileItem` (renamed from `KfPendingItem`):
  ```typescript
  interface KnowledgeFileItem extends QueueItem {
    id: string;          // uses fileId as the queue item id
    projectId: string;
    projectName: string;
    fileId: string;
    fileName: string;
    fileType: string;
    fileSize: number;
  }
  ```
- **Worker**:
  1. Fetch metadata: `net.fetchJson("/backend-api/files/download/{fileId}?gizmo_id={projectId}&inline=false")`
  2. If `file_not_found` → immediate dead-letter (set fail count to max)
  3. Download blob
  4. Store to IDB: `kf/{sanitizedProjectName}/{sanitizedFileName}`
  5. Store `project.json` per project (idempotent — same data on every write)
- **Concurrency**: Default 3, configurable via UI
- **Max retries**: 3
- **Initial items**: `S.progress.knowledgeFilesPending` (populated by scan)

### Wiring (Push Model)

The chat queue worker directly calls `attachmentQueue.enqueue()`. This is the only coupling between queues. The chat queue receives a reference to the attachment queue at construction time (dependency injection). Queues are otherwise fully independent.

```
┌─────────────┐     enqueue()     ┌──────────────────┐
│  Chat Queue  │ ──────────────→  │  Attachment Queue │
└─────────────┘                   └──────────────────┘
       ↑                                    ↑
  populated by Scan              populated by Chat Queue
```

The knowledge file queue has no dependencies on other queues.

## "Download What's Ready" Feature

- Download button enabled at all times if `accumulatedSize > 0`
- Click triggers `downloadFinalZip()` which generates a ZIP from current IDB contents
- Queues continue running during ZIP generation — `stop()` is NOT called
- IDB handles concurrent read (ZIP generation) and write (queue workers) via transaction isolation
- No tracking of "what was included in each download" — each download is a point-in-time snapshot of IDB contents
- `generateFinalZip` needs no changes — it already iterates IDB cursors
- Worst case: a file being written during ZIP generation might not be included. It will be in the next download.

## Rate Limiting & Shared Resources

- All queues share the same `Net` instance
- 429 backoff in `net.ts` applies globally: when any queue triggers a 429, all requests from all queues back off
- Token refresh (`getToken`) is shared and handled transparently
- `pause` setting (delay between items per worker) is shared across all queues — single value, not per-queue
- With 3 queues x 3 workers = 9 potential concurrent requests at peak. The 429 exponential backoff is the real protection against over-requesting.

## Start/Stop/Rescan Behavior

### Start

1. If no pending items and no scan data → trigger rescan first
2. If backoff carryover → wait
3. Start all three queues concurrently
4. Chat queue initialized from `S.progress.pending`
5. Knowledge file queue initialized from `S.progress.knowledgeFilesPending`
6. Attachment queue starts empty, receives items from chat queue workers as conversations are processed

### Stop

Single button stops all three queues via shared AbortSignal. Workers finish their current item and exit. All queue state is persisted to IDB.

Individual queue stop is NOT supported (deferred as nice-to-have).

### Rescan

Same as today. Queues must be stopped first.

### Completion Detection

Each queue fires `onDrained` when it empties. A coordinator tracks drain state:

- Chat drained + Attachment drained + Knowledge drained → status: "All done", fire `onExportComplete`
- Chat drained only → status: "Chats done. Files still downloading..."
- Note: the attachment queue may drain and refill multiple times (empty → chat enqueues more → drains again). Only the final drain (when chat queue is also drained) signals true completion.

## Migration (v2 → v3)

When loading persisted state with `v === 2`:

1. **New fields**: `progress.filePending = []`, `progress.fileDead = []`, `progress.fileFailCounts = {}`, `progress.fileDoneCount = 0`
2. **Rename KF fields**: `kfExported` → `knowledgeFilesExported`, `kfPending` → `knowledgeFilesPending`, `kfDead` → `knowledgeFilesDead`, `kfFailCounts` → `knowledgeFilesFailCounts`
3. **Settings**: `settings.chatConcurrency = settings.conc || 3`, `settings.fileConcurrency = 3`, `settings.knowledgeFileConcurrency = 3`. Remove `batch` and `conc`.
4. **Stats**: map `chats` → `chatsExported`, `batchMs` → `chatsMs`, `kfFiles` → `knowledgeFilesDownloaded`, `kfMs` → `knowledgeFilesMs`. Add `filesDownloaded = 0`, `filesMs = 0`. Remove `batches`, `kfBatches`.
5. **Set** `v = 3`

IDB blob stores (`cvz-export-blobs`) need no migration — same object stores.

## Testing Strategy

### Queue Abstraction (unit tests) — most critical

- Basic enqueue → process → done flow
- Concurrency limit respected (N workers max)
- Retry logic: fail < maxRetries → requeue, fail >= maxRetries → dead-letter
- Enqueue while running (parked workers wake up)
- Stop while running (workers complete current item, then stop)
- `setConcurrency` while running (increase/decrease)
- Pause between items
- AbortSignal cancellation
- Empty queue start (workers park, then wake on first enqueue)

### Queue Instances (integration-style unit tests)

- Chat worker: fetches JSON, extracts file refs, enqueues to attachment queue, stores to IDB
- Attachment worker: dedup check, downloads blob, stores to IDB
- Knowledge file worker: adapted from existing tests

### UI Tests

- Per-queue stats rendering (3 rows with progress bars)
- Download button enabled while running
- Concurrency inputs per queue

### Migration Tests

- v2 → v3 state transformation with all field renames and new defaults
