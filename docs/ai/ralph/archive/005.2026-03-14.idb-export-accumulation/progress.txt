## Codebase Patterns

Squash from: f762ac3f7225cd365a2efbeef0bef1b2ba727974
- js-exporter tests use `// @vitest-environment jsdom` + `import "fake-indexeddb/auto"` for IDB tests
- `vi.resetModules()` in `beforeEach` resets module cache but NOT the fake-indexeddb data; must clear IDB stores explicitly between tests
- fake-indexeddb does not preserve `Blob` type through structured clone in jsdom; values come back as plain objects. Use `value instanceof Blob ? value.size : new Blob([value]).size` for defensive size calculation
- IDB helper modules follow the pattern: module-level `_db` variable, `init*()` export to open DB, all methods are no-ops when `_db` is null
- Each IDB operation opens a short-lived transaction (never held across async boundaries)
- `npm install` is required in the js-exporter worktree before running tests
- `ExportBlobStore` is injected into `createExporter` via the `exportBlobStore` dep in `ExporterDeps`, not imported directly -- enables clean test mocking
- `ExportBlobStoreApi` interface is exported from `exporter.ts` for typing the dep
- `StreamingZip` uses data descriptors (bit 3) for Blob entries to avoid buffering; Uint8Array entries write CRC/size directly in the local header
- `crc32Update` / `crc32Final` in `crc32.ts` enable incremental CRC computation for streaming
- `dosTimeDate` is exported from `zip-lite.ts` and reused in `streaming-zip.ts`
- `generateFinalZip` takes a `getWritableStream` callback for testability; `downloadFinalZip` wraps it with FSAA (`showSaveFilePicker`)
- `FinalZipBlobStore` interface in `generate-final-zip.ts` requires `iterateConvs` and `iterateFiles` -- separate from `ExportBlobStoreApi` (which only has put/totalSize)
- UI optional deps (`getAccumulatedSize`, `onDownload`, `onReset`) use `?:` in `UIDeps` so tests can omit them; handlers check existence before calling
- `onExportComplete` in `ExporterDeps` is optional (`?:`) and only called when export finishes naturally (not stopped by user)

## 2026-03-14 - US-004

- Implemented `StreamingZip` class at `js-exporter/src/zip/streaming-zip.ts`
- API: `constructor(writable: WritableStream)`, `addEntry(path, data: Uint8Array | Blob)`, `finalize()`
- Uses STORE method (no compression), matching `ZipLite`
- For `Blob` data: uses data descriptors (bit 3 of general purpose flags) to enable true streaming -- writes local header with zero CRC/sizes, streams blob chunks via `blob.stream().getReader()` computing CRC incrementally, then writes data descriptor with actual CRC/sizes
- For `Uint8Array` data: computes CRC directly and writes CRC/sizes in the local header (no data descriptor)
- Central directory entries (~100 bytes each) are the only state accumulated in memory
- Added `crc32Update()` and `crc32Final()` to `js-exporter/src/zip/crc32.ts` for incremental CRC-32 computation
- Files changed: `js-exporter/src/zip/streaming-zip.ts` (new), `js-exporter/src/zip/crc32.ts` (modified), `js-exporter/tests/zip/streaming-zip.test.ts` (new, 11 tests), `js-exporter/tests/zip/crc32.test.ts` (modified, +3 tests)
- **Learnings for future iterations:**

  - ZIP data descriptors (bit 3) allow writing file data before knowing CRC/size, which is required for true streaming of Blob content. The descriptor follows the file data with signature `0x08074b50` + crc32 + compressed size + uncompressed size (16 bytes total).
  - `WritableStream` in test environments works well via `new WritableStream({ write(chunk) {...} })` to collect output for verification.

## 2026-03-14 - US-005

- Implemented `generateFinalZip()` and `downloadFinalZip()` in `js-exporter/src/export/generate-final-zip.ts`
- `generateFinalZip` accepts `getWritableStream` callback for testability; iterates conv store to build `conversations-NNN.json` files (100 per file, 1-indexed, 3-digit zero-padded), then iterates files store writing each entry with its IDB key as the zip path
- `downloadFinalZip` wraps `generateFinalZip` with File System Access API: calls `showSaveFilePicker()` with suggested name `chatgpt-export.zip`, handles AbortError (user cancel) and missing API (non-Chromium) with status messages
- Uses `StreamingZip` for memory-efficient ZIP writing; `finalize()` writes central directory and closes the stream
- ZIP structure matches convoviz's `_SPLIT_FILE_RE` pattern and loader expectations
- Files changed: `js-exporter/src/export/generate-final-zip.ts` (new), `js-exporter/tests/export/generate-final-zip.test.ts` (new, 12 tests)
- **Learnings for future iterations:**

  - `generateFinalZip` collects all conversation JSON strings in memory via `iterateConvs` callback before writing batches. This is necessary because IDB cursor callbacks are synchronous (can't await zip writes inside them). For very large exports (10k+ convs), this could use significant memory. A chunked cursor approach would require multiple IDB transactions.
  - `showSaveFilePicker` throws `AbortError` when the user cancels -- this is different from the API not being available (which is a `TypeError` for calling undefined as function).
